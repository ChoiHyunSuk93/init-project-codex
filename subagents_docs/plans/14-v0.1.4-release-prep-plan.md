# v0.1.4 릴리스 준비 계획

## 배경

현재 저장소는 `main` 브랜치에 있고, 공개 태그는 `v0.1.3`까지 존재한다. 이번 릴리스는 다음 patch 버전인 `v0.1.4`를 위한 최소 준비 작업이다.

## 목표

1. 저장소의 버전 표기를 `v0.1.4`로 맞춘다.
2. 릴리스 준비 변경을 최소화한다.
3. 변경 검증 후 `main` 브랜치에 커밋하고, `v0.1.4` 태그를 만들어 푸시한다.

## 범위

### 수정 대상

- `README.md`
- `README.ko.md`

### 비범위

- `hs-init-project/` 아래의 skill 본문, 템플릿, 스크립트
- 새 changelog 파일 생성
- 릴리스 이외의 문서 개편

## 작업 분해

### Plan A: 버전 문자열 갱신

- `README.md`의 최신 공개 릴리스를 `v0.1.4`로 바꾼다.
- `README.ko.md`의 최신 공개 릴리스를 `v0.1.4`로 바꾼다.
- 현재 검색된 범위상 `v0.1.3` 버전 문자열은 이 두 파일에만 남아 있으므로, 변경 범위도 이 두 파일로 제한한다.

### Plan B: 검증

- 두 README에서 버전 표기가 `v0.1.4`로 바뀌었는지 확인한다.
- 작업 트리에 의도하지 않은 추가 변경이 없는지 확인한다.
- 릴리스 준비가 끝나면 `planner -> generator -> evaluator` 순서대로 통과 여부를 확인하고, pass 후에 commit / push / tag / tag push를 진행한다.

## 수용 기준

- `README.md`와 `README.ko.md`의 최신 공개 릴리스 표기가 `v0.1.4`다.
- `hs-init-project/`에는 이번 릴리스 준비에서 불필요한 수정이 없다.
- 검증 후 `main`에 커밋하고 `v0.1.4` 태그를 만든 뒤 푸시할 수 있는 상태가 된다.

## 다음 단계

- generator는 README 두 파일만 수정한다.
- evaluator는 README 표기와 작업 트리 범위만 재확인한다.
- pass 후 커밋, 푸시, 태그 생성, 태그 푸시를 수행한다.
