# 구현 기록 규칙

## 목적

`docs/implementation/` 아래에 구현 이력을 어떻게 저장하는지 정의한다.

## 카테고리 모델

- 관심사 기반 카테고리 디렉토리를 사용한다.
- 카테고리 디렉토리는 해당 관심사의 첫 기록이 실제로 필요할 때 생성한다.
- 카테고리는 관찰된 source area, architecture boundary, 반복 작업 흐름, 기존 documentation domain을 기준으로 정한다.
- `misc`, `general`, `notes`, `other` 같은 약한 catch-all 카테고리는 피한다.

## 기록 배치

- 초기화 단계에서 빈 카테고리 디렉토리나 placeholder 기록 문서를 미리 만들지 않는다.
- 각 기록은 가장 가까운 기존 카테고리 디렉토리에 넣는다.
- 현재 카테고리로 더 이상 작업을 설명하기 어렵게 되었을 때만 새 카테고리를 추가한다.
- flat layout을 명시적으로 요청하지 않았다면 기록을 `docs/implementation/` 루트에 평평하게 쌓지 않는다.
- 구현 변경이 있을 때마다 해당 구현 기록 문서를 생성하거나 수정한다.
- 선택한 카테고리 안에서는 번호를 순서에 맞게 유지한다.

## 파일 이름

- 순번형 markdown 파일명 `NN-name.md` 형식을 사용한다.
- Korean 모드에서는 제어 파일이 아닌 사람이 읽는 파일명 부분을 한글로 써도 된다.
- 각 category 안에서는 번호를 안정적이고 순차적으로 유지한다.

## 기록 양식

- 각 구현 기록은 최소한 `요약`, `변경 내용`, `이유`, `검증`, `관련 규칙` 섹션을 포함한다.
- 새 기록을 만들 때는 이 최소 구조를 유지한 상태에서 실제 변경 내용으로 채운다.
