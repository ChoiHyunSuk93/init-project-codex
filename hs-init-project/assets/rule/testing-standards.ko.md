# 테스트 표준 규칙

## 목적

이 저장소에서 단위 테스트, E2E 테스트, 검증 기대치를 어떻게 정하고 유지하는지 정의한다.

## 신규 저장소

- 이 문서는 provisional한 규칙으로 시작한다.
- 실제 스택이 정해지기 전에는 특정 unit test 또는 E2E framework를 미리 가정하지 않는다.
- 더 강한 프로젝트 규칙이 생기기 전까지는 다음 기준을 사용한다.
  - 단위 테스트는 가능한 경우 고립된 로직, 데이터 변환, edge case 동작에 사용한다.
  - E2E 테스트는 사용자 핵심 흐름이나 경계를 넘는 동작이 실제로 생긴 뒤 그 흐름을 검증하는 데 사용한다.
  - 변경을 가장 안정적으로 검증할 수 있는 가장 작은 테스트 계층을 우선한다.
- 실제 경로, 명령, 툴링이 구체화되면 일반 규칙을 프로젝트별 규칙으로 교체한다.

## 기존 저장소

- 어떤 테스트 규칙을 authoritative로 보기 전에 관찰된 테스트 디렉토리 구조, naming pattern, 명령, 툴링을 먼저 분석한다.
- 일반 기본값보다 저장소에 이미 있는 테스트 디렉토리, 명령, 이름 규칙을 우선한다.
- 저장소가 이미 단위 테스트와 E2E 테스트를 분리하고 있다면 그 구분을 여기서 문서화하고 따른다.
- 특정 영역에 더 강한 테스트 규칙이 있다면 하나의 평평한 전역 규칙으로 덮지 말고 local rule로 더 좁게 분리한다.

## 기본 기대치

- 동작이 바뀌면 가능하면 가장 관련 있는 최소 자동화 테스트 계층을 추가하거나 수정한다.
- 집중된 단위 테스트가 더 적합한 변경을 E2E 테스트만으로 대체하지 않는다.
- 저수준 로직을 검증해야 하는 변경에 과도한 E2E 범위를 기본값처럼 추가하지 않는다.
- 아직 자동화 테스트 경로가 없다면 짧은 수동 검증 메모를 남기고, 실제 테스트 경로가 생기면 이 규칙에 반영한다.
- 역할 분리 하네스를 따를 때 generator는 단위 수준 검증을 우선하고, evaluator는 대표 사용자 surface 직접 검증을 포함한 strongest feasible user-surface/E2E 검증을 맡는다.
- 대표 사용자 surface가 있으면 evaluator는 브라우저, 시뮬레이터, 런타임, CLI, API 호출처럼 해당 프로젝트 타입의 실제 진입점을 가능한 한 직접 검증한다.
- 대표 사용자 surface를 직접 실행하지 못했다면 이유, 누락된 환경, 대체 검증 한계와 남은 공백을 검증 기록에 남긴다.

## 검증 기록

- 구현 기록에는 어떤 단위 테스트를 추가하거나 실행했는지 적는다.
- 관련이 있다면 어떤 E2E 테스트를 추가하거나 실행했는지도 적는다.
- 테스트를 추가하거나 실행하지 못했다면 그 이유와 대신 사용한 수동 검증 방법을 기록한다.
