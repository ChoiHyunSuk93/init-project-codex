# 개발 표준 규칙

## 목적

이 저장소에서 구현 품질 규칙을 어떻게 정하고 유지하는지 정의한다.

## 신규 저장소

- 일반적인 기본값을 최종 프로젝트 표준처럼 고정하지 않는다.
- 실제 스택, 구조, 툴링 관례가 드러나기 전까지는 다음 최소 기대치만 적용한다.
  - 명확한 이름
  - 집중된 책임
  - 명시적인 데이터 흐름
  - 읽기 쉬운 제어 흐름
  - 올바른 오류 처리
  - 코드, 문서, 검증의 동기화
- 실제 프로젝트 관례가 드러나면 이 문서의 일반 규칙을 관찰된 규칙으로 교체한다.

## 기존 저장소

- 어떤 표준을 authoritative로 보기 전에 관찰된 source layout, naming pattern, 툴링, 자동화, 기존 문서를 먼저 분석한다.
- 저장소 위에 이미 존재하는 프로젝트별 관례를 일반 기본값보다 우선해 규정한다.
- 특정 영역에 더 강한 규칙이 있다면 이 문서나 더 좁은 local rule 문서에 구체화한다.

## 기본 품질 기대치

- 가능하면 함수, 모듈, 파일이 하나의 분명한 책임에 집중하도록 유지한다.
- 과도하게 압축된 영리한 표현보다 읽기 쉬운 제어 흐름을 우선한다.
- 오류는 올바른 경계에서 처리하고, 조용히 실패하는 동작은 피한다.
- 동작이 바뀌면 관련 타입, 스키마, DTO, 인터페이스, 문서도 함께 맞춘다.
- 변경으로 인해 생긴 dead code, 오래된 주석, 명백한 중복은 정리한다.
- 문서, 테스트, 외부에서 보이는 동작이 실제 구현과 어긋나지 않게 유지한다.

## 검증 기대치

- 변경한 영역에 가장 관련 있는 검증을 실행하거나, 실행하지 못했다면 어떤 검증이 필요한지 짧게 남긴다.
- 저장소에 이미 lint, type-check, test, formatting 명령이 있다면 새 규칙을 만들지 말고 기존 명령을 사용한다.
- 자동화된 검증이 아직 없다면 최소한의 수동 검증 메모를 남긴다.

## 지속적인 구체화

- 언어나 프레임워크에 더 강한 프로젝트별 표준이 드러나면 일반 규칙을 관찰된 규칙으로 교체한다.
- 일반 기본값보다 관찰된 프로젝트 관례를 우선한다.
- 단위 테스트와 E2E 테스트의 구체 규칙은 이 문서에 섞어 쓰지 말고 `rule/testing-standards.md`에 둔다.
